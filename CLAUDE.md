# Smart Wallet Tracker

> Опорный документ проекта. Это «основа», к которой обращаемся в каждой сессии:
> видение, модель данных, экраны, требования к Telegram, текущее состояние и план.
> При изменении архитектуры — **сначала обновляем этот файл**, потом код.

---

## 1. Видение продукта

Приложение для **личного учёта финансов**. Максимально простое. Помогает вести три вещи:

1. **Расходы** — сколько и на что потратил.
2. **Доходы** — сколько и откуда заработал.
3. **Состояние и счета** — какой сейчас баланс на банковских счетах / картах / наличных,
   и какое суммарное «состояние» (net worth) на данный момент.

**Принцип проектирования: data-first.** Сначала фиксируем простую модель данных,
от неё выстраиваем дизайн-систему и экраны. Данные простые — три смысловые сущности.

### Способы ввода данных (равноправные)

- **Telegram-бот** — пользователь пишет в чат **текстом** («потратил 500 на кофе»,
  «зарплата 120000»). Бот разбирает текст и создаёт транзакцию.
  - **Парсинг текста — через LLM (Claude API).** Приоритет: «работающе и просто».
    Модель извлекает из свободной фразы сумму, тип (доход/расход), категорию, описание,
    при необходимости валюту. Если категории нет — создаётся на лету (см. раздел 2).
  - **Голосовой ввод (audio → текст) — отложен.** На первом этапе только текст;
    распознавание речи добавим позже отдельным шагом.
- **Telegram Mini App** — пользователь открывает мини-приложение и вносит/смотрит статистику:
  сколько заработал, сколько потратил, состояние счетов.

### Ключевое правило идентичности

**1 Telegram-пользователь = 1 пользователь приложения.**
Все данные принадлежат конкретному пользователю и изолированы по его Telegram ID.

## 2. Модель данных

Пользователь мыслит **тремя сущностями**: Доходы, Расходы, Состояние/Счета.
В реализации доходы и расходы — это одна таблица `Transaction` с полем `type`.

### `User` ✅ *(реализовано, Этап 1)*
- `id` (UUID, PK)
- `telegramId` (bigint, unique) — связь «1 TG-юзер = 1 юзер приложения»
- `username`, `firstName`, `lastName`, `languageCode` (опционально, из Telegram)
- `baseCurrency` (string ISO-4217, default `RUB`) — валюта для подсчёта общего «состояния» и сводной статистики
- `isOnboarded` (boolean, default `false`) — прошёл ли пользователь онбординг (выбор базовой валюты)
- `createdAt`, `updatedAt`

### `Transaction` — доходы и расходы
- `id` (UUID, PK)
- `userId` (FK → User)
- `type` (enum: `income` | `expense`)
- `amount` (decimal 10,2) — в валюте счёта
- `currency` (string ISO-4217) — валюта операции (= валюте счёта)
- `exchangeRate` (decimal) — **снимок** курса `currency → user.baseCurrency` на момент операции
- `amountInBase` (decimal 10,2) — сумма в базовой валюте (= `amount × exchangeRate`), для быстрой агрегации
- `categoryId` (FK → Category)
- `accountId` (FK → Account)
- `description` (string ≤ 500)
- `date` (timestamp)
- `createdAt`, `updatedAt`

### `Account` — банковские счета / состояние
- `id` (UUID, PK)
- `userId` (FK → User)
- `name` (string ≤ 100)
- `type` (enum: `bank` | `card` | `cash`) — тип счёта (не валюта!)
- `currency` (string ISO-4217, default `RUB`) — валюта счёта
- `balance` (decimal 10,2, default 0) — в валюте счёта, пересчитывается автоматически при транзакциях
- `color` (HEX `#RRGGBB`)
- `createdAt`, `updatedAt`

> **«Состояние на данный момент»** = сумма балансов всех счетов пользователя,
> приведённая к `user.baseCurrency` по текущим курсам (балансы счетов хранятся каждый в своей валюте).

### `Category` — справочник категорий
- `id` (UUID, PK)
- `userId` (FK → User) (категории должны быть приватными)
- `name` (string ≤ 100)
- `icon` (emoji)
- `color` (HEX `#RRGGBB`)
- `type` (enum: `income` | `expense`)
- `createdAt`, `updatedAt`

> Категории создаются **на лету** при вводе транзакции (отдельной страницы управления нет, см. раздел 3).

### `ExchangeRate` — кэш курсов валют
- `id` (UUID, PK)
- `base` (string ISO-4217) — базовая валюта котировки
- `quote` (string ISO-4217) — котируемая валюта
- `rate` (decimal)
- `date` (date) — на какую дату курс
- `fetchedAt` (timestamp)
- Уникальность по (`base`, `quote`, `date`).

### `Transfer` — перемещение средств между счетами
- `id` (UUID, PK)
- `userId` (FK → User)
- `fromAccountId` (FK → Account) — счёт-источник
- `toAccountId` (FK → Account) — счёт-приёмник
- `amountFrom` (decimal 10,2) — списано, в валюте источника
- `currencyFrom` (string ISO-4217)
- `amountTo` (decimal 10,2) — зачислено, в валюте приёмника
- `currencyTo` (string ISO-4217)
- `rate` (decimal 18,8) — **снимок** фактического курса перевода = `amountTo / amountFrom`
- `description` (string ≤ 500, опц.)
- `date` (timestamp)
- `createdAt`, `updatedAt`

> Перевод — **не доход и не расход**, в агрегаты доходов/расходов не попадает (отдельная таблица).
> Создание: атомарно `fromAccount.balance −= amountFrom` (валидируется минус) и `toAccount.balance += amountTo`.
> Удаление — откатывает оба баланса (атомарно). Кросс-валютная сумма зачисления считается на клиенте по
> текущему кросс-курсу (через USD-пивот) и **редактируема** — можно вбить фактический курс банка.

**Инвариант баланса:** создание транзакции меняет `account.balance` в валюте счёта
(income +, expense −), удаление — откатывает. Реализовано атомарно через TypeORM-транзакции.
Расход не должен уводить баланс в минус (валидируется на сервере).

## 2.1 Мультивалютность

Реализуем **настоящую мультивалютность** (не только задел).

- **Поддерживаемые валюты на старте (10):**
  - Мировые топ-5: `USD`, `EUR`, `JPY`, `GBP`, `CNY`
  - СНГ топ-5: `RUB`, `KZT`, `UZS`, `BYN`, `AZN`
  - Список держим единой константой/enum (один источник правды на сервере и клиенте), легко расширяется.
- **Источник курсов:** `open.er-api.com` (бесплатно, без ключа, есть RUB и все валюты выше).
- **Кэширование:** курсы сохраняем в таблицу `ExchangeRate`, обновляем **раз в сутки**
  (фоновая задача / cron). API не дёргаем на каждый запрос.
- **Модель конвертации:** при создании транзакции сохраняем **снимок курса** (`exchangeRate`)
  и `amountInBase` на момент операции → историческая статистика стабильна и не «плывёт».
- **Базовая валюта** — у пользователя (`user.baseCurrency`, default `RUB`). В ней считается
  общее «состояние» и сводные отчёты. Балансы счетов приводятся к базовой по **текущим** курсам.
- **Деньги:** единый хелпер `formatMoney(amount, currency)` (клиент) и аккуратная работа с
  `decimal` на сервере. Валюту всегда хранить рядом с суммой.

## 3. Экраны

Видение пользователя — **3 страницы**. Реализовано (Этап 5).

| # | Страница | Маршрут | Назначение | Статус |
|---|---|---|---|---|
| 1 | **Расходы** | `/` | Внести расход + статистика расходов (итог, график, разбивка по категориям, история) | ✅ `Transactions type="expense"` |
| 2 | **Доходы** | `/income` | Внести доход + статистика доходов + история | ✅ `Transactions type="income"` |
| 3 | **Счета** | `/accounts` | Список счетов, балансы, суммарное состояние | ✅ `Accounts` |

**Реализовано:** отдельной страницы категорий **нет**. Категории создаются **на лету** при вводе
транзакции (выбрать существующую или быстро создать новую прямо в диалоге). Страницы `Home`/`Categories`
и `CategoryDialog` удалены; обе страницы расходов/доходов — это один параметризованный компонент
`pages/Transactions.tsx` (тип приходит из маршрута).

Навигация (`BottomNav`): 3 пункта — **Расходы** (`/`) / **Доходы** (`/income`) / **Счета** (`/accounts`).
Ввод транзакции — пошаговый диалог: **счёт → сумма (нумпад, в валюте счёта) → категория** (существующая или
создать на лету). Счёт выбирается первым, потому что он задаёт валюту операции — на экране суммы валюта счёта
показывается рядом с числом (символ из `CURRENCY_META`), чтобы не было неоднозначности «100 чего».

## 4. Технологический стек

**Монорепозиторий:** `apps/server` (бэкенд) + `apps/client` (фронтенд).

### Backend — `apps/server`
- NestJS 10 + TypeScript
- PostgreSQL + TypeORM 0.3 (БД `smart_wallet_tracker`, `synchronize` в dev)
- Валидация: **Zod** через кастомный `ZodValidationPipe` + декоратор `@ZodValidation` (ошибки на русском)
- Модули: `transactions`, `accounts`, `categories` (CRUD-контроллеры + сервисы)
- Env (реально используются): `DATABASE_URL` *(или раздельные `DB_HOST`/`DB_PORT`/`DB_USERNAME`/`DB_PASSWORD`/`DB_DATABASE`)*, `PORT`, `NODE_ENV`, `CORS_ORIGIN`, `TELEGRAM_BOT_TOKEN`, `ANTHROPIC_API_KEY`, `ANTHROPIC_MODEL`. Шаблон — `apps/server/env.example`. *(`DATABASE_URL` имеет приоритет; на Render даётся «Add from Database».)*

### Frontend — `apps/client`
- React 18 + Vite 5 + TypeScript
- UI: **shadcn/ui** (radix-ui) + Tailwind CSS, иконки `lucide-react`, тосты `sonner`
- Данные: **@tanstack/react-query** (хуки `use-transactions`/`use-accounts`/`use-categories`), `axios`
- Стейт: `zustand` (`financeStore`) — содержит мок-данные, для API не используется
- Графики: `recharts`, даты: `date-fns`, формы: `react-hook-form`
- Env (реально используется): `VITE_API_URL` (default `http://localhost:3000`). Шаблон — `apps/client/.env.example`.

### API (текущее)
- `users`: `GET /me` · `POST /me/onboarding` *(Этап 1)*
- `transactions`: `POST /` · `GET /?type=` · `GET /:id` · `PATCH /:id` · `DELETE /:id` *(снимок курса при создании/редактировании; PATCH — Этап 6)*
- `accounts`: `POST /` · `GET /` · `GET /net-worth` · `GET /:id` · `PATCH /:id` · `DELETE /:id` *(валюта счёта + состояние, Этап 2)*
- `categories`: `POST /` · `GET /` · `GET /:id` · `PATCH /:id` · `DELETE /:id`
- `exchange-rates`: `GET /` (курсы USD→X) · `POST /refresh` *(Этап 2)*
- `transfers`: `POST /` · `GET /` · `DELETE /:id` *(перевод между счетами, атомарный пересчёт балансов; кросс-валюта со снимком курса)*

> Все эндпоинты защищены `TelegramAuthGuard` (глобальный) и изолированы по `userId`.
> Авторизация: заголовок `x-telegram-init-data` (HMAC по bot token) в prod; в dev — обход через фиксированного тестового юзера.

## 6. Конвенции и правила работы

- Весь код — на **TypeScript** (strict). Никаких `any` без явной причины; общие типы
  (валюты, enum'ы, DTO) держим в одном месте и переиспользуем на сервере и клиенте.
- Язык общения и UI-текстов — **русский**. Сообщения об ошибках валидации — на русском.
- Деньги — `decimal(10,2)` на сервере; всегда хранить **валюту рядом с суммой**;
  на клиенте форматировать через единый `formatMoney(amount, currency)`.
- Любая мутация транзакции обязана сохранять **инвариант баланса** (раздел 2) и фиксировать
  снимок курса (`exchangeRate`, `amountInBase`) на момент операции (раздел 2.1).
- Новые сущности/поля — через TypeORM; в проде использовать миграции (не `synchronize`).
- При добавлении мультипользовательности: **каждый** запрос фильтруется по `userId`
  текущего пользователя; нельзя возвращать чужие данные.
- Изменил модель данных или экраны → **обнови этот файл (CLAUDE.md)**.

## 8. Решения по разработке

- **Порядок:** сначала детальный пошаговый план, потом код.
- **Общие типы:** пакет `packages/shared` в монорепо — типы, enum'ы (валюты, типы счетов/транзакций),
  Zod-схемы. Единый источник правды, переиспользуется сервером и клиентом.
- **Telegram-авторизация в dev:** dev-обход (фиксированный тестовый пользователь без Telegram)
  + реальная валидация `initData` (HMAC по bot token) в prod.
- **Текущая БД:** данные тестовые → **пересоздаём схему с нуля** (с `User`, `currency` и пр.),
  миграцию старых данных не делаем.

## 10. Деплой (см. DEPLOY.md)

- **Бэкенд + БД** → Render (`render.yaml`): build `npm install && npm run build:server`, start `npm run start:server`.
- **Клиент** → Vercel (`vercel.json`): Root Directory = **корень репо**, build `npm run build:client`, output `apps/client/dist`, env `VITE_API_URL`.
- **Прод-инвариант:** `NODE_ENV=production` отключает dev-обход → нужен валидный Telegram `initData`; у бэкенда и у бота-владельца Mini App — **один `TELEGRAM_BOT_TOKEN`**. `CORS_ORIGIN` = URL клиента.
- **Mini App:** в @BotFather кнопка-меню бота → URL клиента (Vercel). Бот (long-polling) живёт внутри NestJS; на free-tier Render засыпает.
